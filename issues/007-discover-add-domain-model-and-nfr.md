# 007 - discover 应增加领域模型和非功能需求产出

## 来源

`.claude/commands/discover.md` Layer 2 / Layer 3 + `discover.json` 产物结构

## 问题描述

discover 产出的颗粒度与 spec 输入需求之间存在断层：

### 断层 1：缺少领域模型

discover 定义了功能和用户故事，但没有识别核心领域对象及其关系。spec 阶段需要设计数据模型（实体、字段、类型、关系），却没有上游输入，只能凭空推断。

例如：discover 说"用户可以管理任务"，但"任务"有哪些核心属性、和"项目"是什么关系、有没有"子任务"概念——全靠 spec 猜，不同理解会导致完全不同的设计。

### 断层 2：非功能需求几乎空白

性能目标、安全要求、并发规模等对架构设计影响巨大，但 discover 只在 constraints 里粗略触及，spec 缺乏决策依据。

## 期望行为

在 discover 阶段增加两块轻量级产出：

### 1. 核心领域模型
- 识别核心领域对象（如：用户、任务、项目）
- 定义对象之间的关系（如：项目包含多个任务，用户拥有多个项目）
- 不需要字段级别的细节，只需实体 + 关系
- 写入 `discover.json` 的 `domain_model` 字段

### 2. 非功能需求
- 性能目标（响应时间、吞吐量）
- 安全要求（认证方式、数据加密等）
- 规模预期（用户量、并发量、数据量）
- 写入 `discover.json` 的 `non_functional_requirements` 字段

## 不应上移到 discover 的内容

- 模块分解、接口设计、依赖图 — 这些是 spec 的设计职责
- 字段级数据模型 — 过于细节，属于 spec 阶段
