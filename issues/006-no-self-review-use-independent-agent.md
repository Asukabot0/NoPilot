# 006 - 框架级原则：生成与评审分离 + 统一迭代验证循环

## 来源

- `.claude/commands/discover.md` Layer 3 — 6Cs Assessment（第152-162行）
- 同理适用于 spec、build 等所有阶段的 review 环节

## 问题描述

当前 discover Layer 3 的 6Cs 质量评估是由 discover 自己内联执行的（自己生成需求，自己评分），Critic agent 只是事后复核。这存在"自己给自己打分"的根本缺陷：

- 生成者天然倾向于认为自己的产出是合格的（self-approval bias）
- Critic 事后复核只是补救，无法改变第一遍自评已经影响了流程推进的事实
- 这个问题不仅存在于 discover，所有阶段涉及 review/评估的环节都有同样的隐患

## 期望行为

- **生成和评审严格分离**：生成内容的 agent 不得评估自己的产出
- 所有 review/评估环节必须由独立的 agent 执行
- 这条原则 + 迭代验证循环作为**框架级设计约束**，统一适用于所有 review 环节：

### 适用范围

| 阶段 | review 环节 | 当前问题 |
|------|------------|----------|
| discover | 6Cs 质量评估 | 内联自评 |
| discover | Critic 复核 | 仅事后补救 |
| spec | backward verification（验收标准 vs spec） | 需确认是否自评 |
| spec | Supervisor 全局一致性检查 | 需确认流程 |
| build | 场景验证（代码 vs spec） | 需确认流程 |
| build | auto-acceptance 验证 | 需确认流程 |

## 修改后的迭代验证流程

实际使用中验证了此问题：discover 自评 6Cs 通过，但独立 agent 重新评估后不通过。修改后的产出同样不能由修改者自评，需要持续由独立 agent 验证直到通过。

### 迭代验证流程

```
独立 agent A 评估 → 不通过 → 执行 agent 修改
    ↓
独立 agent B 评估（新实例，不带 A 的上下文，避免锚定效应）
    ↓
┌─ 通过 → 继续
└─ 不通过 → 执行 agent 再改 → 独立 agent C 评估
                                    ↓
                              ┌─ 通过 → 继续
                              └─ 仍不通过 → 达到上限，提交人类决策
```

### 关键约束

- 每轮评估的 agent 必须是**新的独立实例**，不带之前对话上下文，避免前一轮评估者的偏见锚定
- 迭代上限应**根据任务复杂度浮动**，而非固定值：
  - 简单 review（如单个需求的 6Cs）→ 上限 3 次
  - 中等 review（如模块级 backward verification）→ 上限 5 次
  - 复杂 review（如全局一致性、跨模块验证）→ 上限 7-10 次
  - 上限可由框架根据 review 涉及的范围和影响面自动评估
- 达到上限仍不通过时，不应直接提交人类，先评估原因：
  - **问题严重性在收敛**（每轮问题数减少或严重性降低）→ 适当延长循环，模型在逼近正确答案
  - **问题在发散**（修改引入新问题，总量不减反增）→ 可能是模型能力不足导致的"按下葫芦浮起瓢"，考虑换更强的模型重试
  - **问题反复震荡**（A→B→A→B 循环）→ 需求本身有歧义或存在矛盾约束，提交人类决策

## 改进方向

- 移除 discover Layer 3 中的内联 6Cs 自评
- 将所有 review 职责统一交给 Critic 或独立 review agent
- 在 workflow.json 或框架级文档中明确"生成与评审分离"原则
- 审查 spec、build 中是否存在类似的自评环节，统一修正
- 实现迭代验证流程，每轮用新 agent 实例评估，设最大迭代上限
