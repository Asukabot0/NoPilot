# 005 - discover 阶段应帮用户提炼设计哲学

## 来源

`.claude/commands/discover.md` Layer 2 / Layer 3 阶段

## 问题描述

当前 discover 流程产出的是具体的需求、功能列表、技术方向，但缺少对产品**设计哲学**的提炼。

设计哲学是产品的核心信念和原则，例如：
- "人类是决策者，不是执行者"（NoPilot 自身的哲学）
- "约定优于配置"（Rails 的哲学）
- "做一件事并做好"（Unix 哲学）

没有设计哲学，后续 spec 和 build 阶段遇到模糊决策时缺乏判断依据，容易导致产品设计不一致。

## 期望行为

- 在 discover 过程中（可能是 Layer 2 或 Layer 3），从用户的讨论中提炼出 3-5 条设计哲学
- 设计哲学应该来源于用户表达的偏好、取舍、价值观，而非 AI 硬套模板
- 写入 `discover.json` 作为下游阶段的决策依据
- spec/build 遇到歧义时可以回溯设计哲学来做判断

## 改进方向

- 在 `discover.json` 中增加 `design_philosophy` 字段
- 提炼时机：可以在 Layer 2 MVP 讨论后、Layer 3 需求锁定前
- 提炼方式：从用户的取舍决策中归纳（"你选择 X 而非 Y，说明你重视…"）
- 需要用户确认，而非 AI 单方面总结
