# 008 - 向人类提问前先进行 agent 共识机制

## 来源

全流程（discover / spec / build 所有需要人类确认的环节）

## 问题描述

当前各阶段 agent 遇到歧义或需要决策时直接抛问题给人类，导致：
- 人类被频繁打扰，很多问题 agent 之间其实可以自行解决
- 提问质量参差不齐，有时问题很模糊
- 人类需要自己思考所有上下文才能做决策

## 期望行为：Agent 共识机制

### 流程

```
执行 agent 遇到歧义
    ↓
拉起咨询 agent（锚定设计哲学 + 第一性原则）
    ↓
┌─ 意见一致 → 按分级机制处理
│
└─ 意见不一致 → 再进行一轮讨论尝试收敛
                    ↓
              ┌─ 收敛成功 → 按分级机制处理
              │
              └─ 仍不收敛 → 带双方观点 + 分歧点提交人类决策
```

### 咨询 agent 的决策依据

- **设计哲学为核心**：回溯 discover 阶段提炼的设计哲学作为决策锚点
- **第一性原则**：不做表面妥协，从根本原理出发判断
- 不是"互相迁就求和"，而是"回到根本原则来判断哪个更对"

### 分级机制

- **低影响**（技术实现细节）→ agent 共识即可自动通过，记录到 decision log
- **中影响**（影响架构但不影响产品方向）→ agent 共识后通知人类，无异议则通过
- **高影响**（涉及产品方向、用户可见行为）→ 即使 agent 共识也必须人类确认

### 风险防控

- 两个 agent 都是 LLM，可能存在系统性盲点导致"共识但都错"
- 高影响决策的人类确认是不可跳过的安全阀
- 所有共识决策必须记录到 decision log，便于事后审计
