# 013 - Supervisor 和 Critic 的 prompt 过于简单，需要系统化

## 来源

- `.claude/commands/supervisor.md`
- `.claude/commands/critic.md`

## 问题描述

当前两个核心守卫 agent 的 prompt 过于简略：

### Supervisor 的问题

- 只有 3 个笼统的检查问题（意图对齐、复杂度增长、约束合规），缺少系统化的评估方法论
- 没有定义评估的量化标准或分级体系
- 没有定义如何处理"部分对齐"的灰色地带
- 缺少对不同阶段产物的差异化评估策略
- 缺少 drift 的具体模式识别（常见的偏移类型有哪些）

### Critic 的问题

- 三个阶段的 review 逻辑虽然分别定义了，但缺少统一的评估框架
- self-fix 上限固定为 2 次，与 006 的浮动上限不一致
- 缺少评估的优先级排序（哪些问题 block，哪些是 warn）
- 缺少对 AI 常见偏差模式的系统性检查清单（不只是"检查 self-approval bias"一句话）
- 缺少评估结果的置信度标注

## 期望行为

### Supervisor 应增强的方面

- **系统化的 drift 检测框架**：定义常见 drift 模式（scope creep、gold plating、技术驱动偏移、需求稀释等）
- **量化评估维度**：不只是 aligned/drifted 二元判断，提供偏移程度评分
- **分阶段评估策略**：discover/spec/build 各阶段的 coherence 检查侧重点不同
- **历史决策链追溯**：结合 decisions.json 审查决策链的累积偏移
- **设计哲学合规性**：检查产出是否符合 discover 阶段提炼的设计哲学（关联 005）

### Critic 应增强的方面

- **统一评估框架**：跨阶段一致的评估方法论，而非三段各写各的
- **AI 偏差模式清单**：系统性定义 LLM 常见的生成偏差（过度工程、乐观估计、遗漏负向路径、概念混淆等）
- **评估结果置信度**：标注每个判断的置信度，低置信度的判断建议人类复核
- **与迭代验证循环对齐**：self-fix 上限改为浮动，与 006 一致
- **严重性分级标准**：明确定义 block vs warn 的判断依据

## 关联

- 005（设计哲学）— Supervisor 应检查设计哲学合规性
- 006（生成与评审分离 + 迭代验证循环）— Critic 的 self-fix 机制需要对齐
