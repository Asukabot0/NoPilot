# NoPilot 路线图

[English](../../ROADMAP.md)

## V1 — 核心管线（当前版本）

**状态：** 已实现。采用统一 skills 分发，支持 Claude Code / Codex / OpenCode，当前公开工作流仍以 Greenfield 为主。

**已交付：**
- 三阶段工作流：`/discover` → `/spec` → `/build`
- /discover 三层收敛漏斗（Step 0 + Layer 1/2/3）
- /spec 受约束的设计展开
- /build TDD 实现 + tracer bullet + 自动验收
- Supervisor Agent（意图守护者）+ Critic Agent（独立质疑者）
- 核心护栏：反向验证、全局一致性检查、自动验收
- 增强护栏：tracer bullet（开启）、变异测试（关闭）、多次采样 6Cs（关闭）
- Full/lite 模式 + AI 推荐模式选择
- 分级异常处理（L0/L1/L2/L3）
- 回溯安全（上限 + 循环检测）
- workflow.json 状态机流控
- 结构化 JSON 制品 + 跨阶段追溯（REQ → MOD → TEST/PROP）

**已知限制：**
- 仅 Greenfield——不支持已有代码库
- 回溯时全量重跑下游阶段（无增量更新）
- 所有护栏依赖同模型自评（Critic 用独立会话但同模型）
- 无 MCP/脚本强制约束层（对指令遵循弱的模型）
- 尚未对所有工作流制品启用统一自动 JSON Schema 校验
- 大项目的上下文窗口压力（适配器须管理，但无配套工具）
- 无跨项目持久化记忆

## V1.5 — 可用性与入口

**目标：** 降低实际使用摩擦。解决两个最大缺口：简单项目流程过重、不支持已有项目。

### 真正轻量的 Lite 模式
- [ ] Lite 模式下制品 schema 降级（功能列表替代完整 EARS，不变量可选）
- [ ] Lite discover 阶段减重：为「方向已明确」场景精简 Layer 3 步骤（[#17](https://github.com/Asukabot0/NoPilot/issues/17)）
- [ ] Lite 模式 Layer 1/2 搜索改为推荐而非必选
- [ ] Lite /spec：同会话 Critic，不启动独立会话
- [ ] Lite /build：tracer bullet 和自动验收使用简化检查
- [ ] AI 模式推荐使用明确的启发规则（约束数量、功能数量、平台复杂度）

### Brownfield 支持
- [ ] Feature 级制品隔离：每个 feature 独立目录 `specs/<feature-slug>/`，`.active` 指针标记当前上下文（[#17](https://github.com/Asukabot0/NoPilot/issues/17)）
- [ ] `/discover` 读取已有代码库作为上下文（文件结构、已有 API、技术栈检测）
- [ ] Layer 1 跳过方向发散——方向就是"扩展这个已有项目"
- [ ] Layer 2 在已有架构上下文中呈现功能（什么能直接加、什么需要重构）
- [ ] `/spec` 读取已有代码，避免模块冲突，复用已有模式
- [ ] `/build` 生成与已有代码库集成的代码（import、规范、测试框架）
- [ ] 回溯重跑只影响新增/变更的制品，不影响已有代码

### 搜索加固
- [ ] 搜索失败时优雅降级：`grounding: "ai_judgment_only"` + 用户通知
- [ ] 搜索质量评分——低质量结果标记而非静默消费
- [ ] 所有模式下搜索均为推荐（非必选），支持显式跳过

### 前置环境检查
- [ ] `/spec` 完成时触发环境就绪检查（API key、数据库、所需 CLI）
- [ ] 缺失依赖在 `/build` 启动前暴露，而非在 L0 异常时才发现

## V2 — 可靠性与效率

**目标：** 让管线可靠到足以用于正式项目。降低回溯成本，加入多模型验证，强制制品契约。

### 增量回溯
- [ ] 影响分析：需求变更时追踪受影响的模块和测试
- [ ] `/spec` 只重新生成受影响模块，保留未变更模块
- [ ] `/build` 只重跑受影响模块的 TDD 循环，保留已通过模块
- [ ] 契约修订自动传播：变更向下游流动，无需全量重跑

### MCP/脚本强制约束层
- [ ] 可选 MCP Server：读取 workflow.json，程序化执行状态机转换
- [ ] 所有制品类型的 JSON Schema 验证脚本
- [ ] 制品写入时自动校验——拒绝格式错误的制品
- [ ] 状态转换验证——无论模型是否遵守，阻止非法转换
- [ ] 面向指令遵循弱的模型设计（开源 LLM、小模型）

### 多模型验证
- [ ] Critic Agent 可选使用与生成模型不同的模型
- [ ] Supervisor Agent 可选使用不同模型
- [ ] workflow.json 配置：`agents.critic.model`、`agents.supervisor.model`
- [ ] 彻底打破同模型偏差（不只是独立会话，而是独立模型）

### 多次采样 6Cs
- [ ] 启用 `multi_sample_6cs` 护栏：AI 对每条需求评估 N 次
- [ ] 跨采样 pass/fail 不一致 → 标记为低置信度
- [ ] workflow.json 可配置采样次数

### 上下文管理
- [ ] 按字段定义制品摘要策略（可压缩 vs 硬约束字段）
- [ ] 适配器侧上下文预算管理工具
- [ ] 大制品分块：上下文紧张时按模块分段读取 spec.json

### 统一制品 Schema 校验
- [ ] 统一梳理 `schemas/` 中现有工作流制品 Schema 覆盖范围
- [ ] Schema 与 workflow.json 同步版本化
- [ ] 将校验集成到所有制品写入流程

## V3 — 智能与学习

**目标：** NoPilot 从经验中学习。跨项目模式、spec 漂移检测、更丰富的验证。

### 记忆系统
- [ ] 跨项目经验存储：成功/失败的技术选型、常见模式、架构偏好
- [ ] 工作流定义层定义记忆接口（"记住什么"）
- [ ] 存储/检索由运行时适配器实现（"怎么存"）
- [ ] `/discover` Layer 2 推荐技术方向时参考过往项目决策
- [ ] `/spec` 设计相似系统时参考过往模块模式
- [ ] 记忆衰减：旧记忆权重降低，被矛盾的记忆自动修剪

### Spec 漂移检测
- [ ] 监控代码变更与 spec.json 的偏差——检测实现何时偏离 spec
- [ ] 通过适配器 hook 在 git commit 或文件保存时触发
- [ ] 漂移报告：哪些模块偏离、哪些 spec 约束被违反
- [ ] 建议更新 spec 或纠正代码

### 变异测试
- [ ] 启用 /build Step 5 的 `mutation_testing` 护栏
- [ ] AI 对自己的代码做变异（故意注入 bug）
- [ ] 检查测试是否能捕获变异
- [ ] 未捕获的变异 → 测试质量缺陷 → 生成补充测试
- [ ] 可配置变异数量和策略

### 动态约束维度
- [ ] `/discover` Step 0 的约束维度在运行时可扩展
- [ ] AI 可基于项目上下文建议新的约束维度
- [ ] 用户可在 discovery 过程中添加自定义维度而无需重启

### 更丰富的追溯
- [ ] 验收标准 → 接口级映射（不只是模块级）
- [ ] 测试推导链可视化（需求 → 标准 → 测试 → 代码）
- [ ] 覆盖度缺口分析 + 建议补充测试

## V4 — 平台扩展

**目标：** NoPilot 跑出 Claude Code。iOS 远程 Agent、并行执行、多 LLM 后端。

### iOS 运行时适配器
- [ ] Swift 编排器读取 workflow.json 并执行状态机
- [ ] 远程 Agent 执行：iOS App 向服务器端 LLM Agent 发送任务
- [ ] 多 LLM 后端：将阶段/Agent 路由到不同提供商（Claude、GPT、Gemini、开源）
- [ ] 异步执行：启动 `/build`，关闭 App，完成或 L2 暂停时收到通知
- [ ] 需要人类决策的检查点通过推送通知

### 并行模块执行
- [ ] 无依赖关系的独立模块并行构建
- [ ] 需要从 spec.json 接口自动生成 API Mock（V2 schema 的前置依赖）
- [ ] 并行模块之间的契约测试（集成前）
- [ ] Critic/Supervisor 并行执行（设计上已独立）

### 插件架构
- [ ] 自定义阶段插入：在 discover/spec/build 之间或 build 之后添加阶段
- [ ] 自定义 Agent 插入：在 Supervisor/Critic 之外添加领域特定 Agent
- [ ] 自定义护栏插件：在阶段边界添加用户定义的检查
- [ ] 阶段跳过/替换：将 /build 换成外部 SDD 工具（BMAD、Kiro 等）

### Web 仪表板（可选）
- [ ] 制品可视化：需求 → 模块 → 测试追溯图
- [ ] 管线状态：当前阶段、当前状态、阻塞原因
- [ ] 决策历史：所有人类决策和 AI auto_decisions 的时间线视图
- [ ] 回溯成本估算器：确认回溯前预测重跑时间

## 版本策略

- **大版本**（V1、V2、V3、V4）：改变工作流定义层的新能力
- **小版本**（V1.5）：现有工作流模型内的可用性改进
- **workflow.json `version` 字段**跟踪 schema 兼容性
- 向后兼容：新适配器必须支持旧版 workflow.json
